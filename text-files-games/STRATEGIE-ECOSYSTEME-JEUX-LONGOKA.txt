STRATÉGIE — Écosystème jeux Longoka (longoka-games ↔ longoka)
================================================================
Dernière mise à jour : 2026-04-08
Objectif : une référence courte pour que génération, sync, vitrine web et livres papier restent alignés.

────────────────────────────────────────────────────────────────
1. TYPES DE PACKS GÉNÉRÉS (backend Java — référence unique)
────────────────────────────────────────────────────────────────

Le moteur premium (exports batch / scripts Maven) peut produire des fichiers
*.json par type de jeu. Noms de schéma / suffixes habituels :

  • wordsearch   — mots mêlés (socle livre + web)
  • crossword    — mots croisés (livre + web)
  • arrowword    — mots fléchés (souvent dérivé des grilles crossword ; très adapté web)
  • domino       — dominos morphologiques
  • memory       — memory / paires
  • scrabble     — construction de mots type Scrabble
  • anagram      — anagrammes morpho

Tout pack doit rester compatible avec le schéma attendu côté site (Nuxt) :
champs stables, meta explicite (gameType, langue, difficulté, book si livre).

────────────────────────────────────────────────────────────────
2. RÔLES DES DEUX DÉPÔTS
────────────────────────────────────────────────────────────────

longoka-games (ce dépôt)
  — Bases lexicales (Lexikongo / Lexilingala) + générateurs Java
  — Export JSON vers backend-java/target (souvent weekly/ ou biweekly/)
  — Scripts JSX / chaîne InDesign pour PRINT

longoka (site Nuxt)
  — Copie synchronisée des JSON dans server/data/games (+ bundle Nitro)
  — UI jouable, APIs, règles de visibilité « public » vs « cours »
  — Pas de logique lexicale lourde : consommation de packs figés

Règle d’or : une évolution de format JSON se conçoit d’abord ici, puis on
vérifie le lecteur côté longoka (composants + detectPackType).

────────────────────────────────────────────────────────────────
3. PIPELINE DE LIVRAISON (bout en bout)
────────────────────────────────────────────────────────────────

A. Export DB → JSON (local ou CI longoka)
   — run-export-all-premium.sh (cadence weekly | biweekly, MEANING_LANG fr/en)
   — Artefacts : backend-java/target/[cadence]/*.json

B. Sync vers le site
   — script longoka : scripts/sync-games-packs.mjs
   — Copie JSON + JSX utiles vers server/data/games
   — Puis generate-games-bundle.mjs pour embarquer la liste des packs

C. Déploiement
   — Commit des JSON + games.generated.mjs ; build / mise en prod

D. Secrets & accès
   — Bases lexicales : secrets LEX_KG_* / LEX_LN_* (GitHub Actions)
   — Dépôt longoka-games privé : token LONGOKA_GAMES_CHECKOUT_TOKEN si besoin

────────────────────────────────────────────────────────────────
4. STRATÉGIE CANAUX : PRINT vs WEB vs « LES DEUX »
────────────────────────────────────────────────────────────────

• Livre papier (priorité produit documentée : mots mêlés, puis mots croisés)
  — JSON « book » volumineux (ex. 48 grilles), mise en page InDesign
  — Ne pas confondre avec les packs « preview » web courts

• Web (longoka)
  — Tous les types listés en §1 peuvent être joués si le front les supporte
  — arrowword / wordsearch / crossword sont les plus naturels pour le trafic

• Métadonnées cibles (meta.targets : web, jsx, indesign, etc.)
  — Utiles pour documenter l’intention ; le site peut un jour filtrer la
    rotation sur ces flags. Tant que ce n’est pas codé, la vérité reste le
    nom de fichier + meta + releaseStatus.

────────────────────────────────────────────────────────────────
5. STRATÉGIE PRODUIT : PUBLIC (VITRINE) vs COURS (APPRENANTS)
────────────────────────────────────────────────────────────────

Principe : le public voit moins et plus tard que les inscrits aux cours.

Implémentation côté longoka (server/utils/learnerGamesPacks.js) — à respecter
lorsqu’on nomme ou qu’on balise les packs :

  • Packs « livre » reconnus via le motif de fichier -book. (kg / ln)
  • Statuts : draft / private / for-sale-only → jamais public
  • Âge des packs et des livres : variables d’environnement (GAMES_PUBLIC_*,
    GAMES_COURSE_*) — le public a des délais plus longs (ex. Lingala)
  • Rotation : vitrine = extraits d’anciennes séries de livres pour ne pas
    cannibaliser le dernier livre en vente ; cours = règles distinctes
    (packs plus récents possibles, autre fenêtre)

Pour le teaser landing (Lingala retardé) : GAMES_PUBLIC_LN_TEASER_START dans
.env du site.

En cas de doute produit : ajuster les env sur longoka, pas le JSON source,
sauf si le pack doit être explicitement « jamais web » (meta + statut).

────────────────────────────────────────────────────────────────
6. RÈGLES D’ÉVOLUTION SANS CASSER LE SITE
────────────────────────────────────────────────────────────────

1) Schéma : préférer des champs optionnels plutôt que renommer brutalement.
2) Nouveau type de jeu : ajouter le générateur ici + le rendu + detectPackType
   côté longoka + clés i18n (libellés).
3) Qualité avant sync : pas de doublons par grille, traductions vides,
   mots trop longs pour la taille annoncée (cf. plans livre dans ce dossier).
4) Fichiers de démo : le sync ignore les noms contenant demo/test/sample.
